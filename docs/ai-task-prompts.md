# AI 任务提示模板

本文档用于把需求整理成 AI 编码代理更容易执行和验证的任务输入。它补充 `AGENTS.md` 和 `docs/ai-agent-guide.md`：前者规定代理必须遵守的硬规则，后者说明常见改动的落地路径，本文档则帮助用户把任务说清楚。

使用这些模板时，不需要机械填写所有字段。高风险、涉及数据一致性、安全、权限、外部依赖或状态流转的任务应尽量完整；低风险文档、文案或局部样式调整可以使用快速任务模板。

## 通用任务模板

```text
任务背景：
【描述业务背景、用户场景、当前问题】

目标：
【描述完成后用户应该能做什么】

不允许发生的情况：
【列出禁止状态、错误行为、安全边界】

不变量：
【列出必须始终成立的规则】

唯一性规则：
【说明用户、订单、文件、会话等实体如何判定为同一个】

状态流转：
【说明状态可以如何变化，哪些变化非法】

验收条件：
1. 【正常路径】
2. 【边界条件】
3. 【错误路径】
4. 【权限路径；不涉及权限时可删除】
5. 【回归路径】

测试要求：
【说明需要单元测试、接口测试、组件测试、端到端测试或手工检查】

验证命令：
【列出项目实际命令，例如 make check、make test、make vet、make compose-check】

约束：
- 保持最小修改。
- 遵循项目已有模式。
- 不引入无关依赖。
- 文档和注释使用中文。
- 最终说明修改内容、验证结果和剩余风险。
```

## 快速任务模板

适用于小改动、低风险修复、局部文档更新或不改变运行行为的配置说明调整。

```text
请完成以下小范围变更：

目标：
【一句话说明】

验收条件：
1. 【可验证条件 1】
2. 【可验证条件 2】
3. 【可验证条件 3】

要求：
- 先阅读相关代码或文档。
- 保持最小修改。
- 不做无关重构。
- 如涉及 Node 命令，先加载 nvm。
- 修改后运行相关验证命令。
- 最终说明修改内容、验证结果和剩余风险。
```

## Bug 修复模板

```text
请修复以下 bug，并使用测试锁定问题：

现象：
【用户看到的问题或测试失败信息】

复现步骤：
1. 【步骤 1】
2. 【步骤 2】
3. 【步骤 3】

期望结果：
【正确行为】

实际结果：
【错误行为】

影响范围：
【受影响的用户、页面、接口、数据或流程】

要求：
- 先复现问题。
- 先写一个失败测试证明问题存在。
- 说明根因，再做最小修复。
- 修复后运行失败测试和相关回归测试。
- 不要通过删除测试、放宽断言或静默吞错来让测试通过。
```

## 新功能模板

```text
请实现以下新功能：

业务目标：
【用户价值和业务目标】

用户流程：
1. 【用户动作】
2. 【系统反馈】
3. 【完成状态】

不变量：
- 【必须始终成立的规则】

约束：
- 【系统必须禁止的输入、状态或行为】

失败模式：
- 【依赖失败时怎么办】
- 【权限不足时怎么办】
- 【重复提交时怎么办】
- 【并发冲突时怎么办】

验收条件：
1. 【正常路径】
2. 【非法输入】
3. 【权限不足；不涉及权限时可删除】
4. 【依赖失败】
5. 【回归路径】

实现要求：
- 先阅读现有实现和测试。
- 先补测试，再实现。
- 使用现有架构和组件。
- 保持改动范围最小。
- 修改后运行相关验证命令。
```

## 代码审查提示

```text
请以代码审查视角检查本次变更，优先指出 bug、回归风险、安全问题、数据一致性问题、并发问题和缺失测试。

请重点检查：
- 需求是否被完整实现。
- 验收条件是否都有测试或手工验证。
- 不变量是否被代码、数据库或测试保护。
- 唯一性规则是否正确。
- 权限和认证是否存在越权风险。
- 错误处理是否覆盖失败路径。
- 是否引入不必要的耦合、依赖或状态。
- 是否有敏感信息泄露风险。
- 是否存在未覆盖的回归路径。

输出格式：
1. 先列问题，按严重程度排序。
2. 每个问题说明风险和建议修复方式。
3. 如果没有发现问题，明确说明剩余测试风险。
```

## AI 自检清单

AI 在最终回复前应自检：

```text
- 我是否先理解了现有项目结构？
- 我是否说明了关键假设？
- 我是否保护了不变量和唯一性规则？
- 我是否覆盖了与本次变更相关的错误路径和权限路径？
- 我是否避免了无关重构？
- 我是否没有擅自引入依赖？
- 我是否运行了相关验证？
- 我是否说明了无法验证的部分？
- 我是否明确了剩余风险？
```

## 不接受的行为

以下行为需要人中止或要求 AI 重做：

```text
- 没读代码就直接大改。
- 只实现正常路径。
- 自动发明不存在的 API、字段或配置。
- 为了通过测试而删除测试或放宽关键断言。
- 静默吞掉错误。
- 擅自引入新依赖、换架构、提交、推送、发布或创建 PR。
- 忽略权限、唯一性、状态流转和失败模式。
- 最终只说“完成了”，但没有验证结果。
```

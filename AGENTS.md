# AI 开发代理工作指令

## 适用范围

- 本文件约束 Codex 或其他 AI 编码代理在本仓库中的默认工作方式。
- 用户的显式要求优先级最高；如果用户要求与本文件冲突，先遵循用户要求，并说明可能风险。
- 更靠近当前工作目录的 `AGENTS.md` 或 `AGENTS.override.md` 可以覆盖本文件规则。

## 基本原则

- 修改前先理解现有代码、测试、文档、Compose 配置和项目约定。
- 不要迎合用户结论；发现错误假设、技术风险或更优方案时，直接说明依据。
- 修改范围保持最小，避免无关重构、无关格式化和元数据噪音。
- 不要回滚、覆盖或删除用户已有改动，除非用户明确要求。
- 文档和注释使用中文；代码命名遵循 Go 和本项目已有风格。
- 能用项目已有工具、框架和辅助函数解决的问题，不另起一套实现。

## 信息时效

- 涉及依赖版本、API 行为、产品状态、运行环境、软件库更新等时效性信息时，必须联网确认。
- 安装、添加或升级项目依赖前，先查询依赖的最新稳定版本和兼容性要求。
- Docker 镜像必须固定具体 tag，不使用 `latest`。
- 不要用训练知识替代当前事实；不确定时说明不确定性并查证。

## 环境约定

- 当前项目主要运行在 WSL 环境。
- 运行 Node.js、npm、pnpm、yarn、npx 等 Node 相关命令前，必须先显式加载 nvm：

  ```bash
  export NVM_DIR="$HOME/.nvm" && . "$NVM_DIR/nvm.sh"
  ```

- 不要假设非交互式 shell 会自动读取 `~/.bashrc`。
- 当前项目没有 Node 构建步骤；以后新增前端、脚本工具或代码生成器时仍需遵守上面的 nvm 规则。
- 优先使用项目已有的 Go、Docker Compose 和 Makefile 命令，不擅自切换工具链。

## 项目边界

- 本仓库是轻量 Go 微服务基础模板，不是生产平台模板。
- 默认保持单个 Go HTTP 服务、PostgreSQL、Redis、可选 Kafka、可选本地可观测性栈。
- 不要为了单个服务默认添加 API 网关、Kubernetes、认证框架、OpenAPI、数据库迁移工具或重型代码生成器。
- Compose 服务名 `service` 是模板约定，不要在重命名业务项目时机械替换。
- 应用代码只读取最终连接 URL：`DATABASE_URL` 和 `REDIS_URL`；不要在 Go 代码里重新拼接数据库或 Redis 连接串。

## 代码探索

- 搜索文本或文件优先使用 `rg` 或 `rg --files`。
- 需要修改文件前先查看 `git status --short`，区分用户已有改动和本次计划改动。
- 修改前阅读相关代码、测试、配置和调用方。
- 不要只根据文件名或局部片段猜测系统行为。
- 复杂问题先定位事实链路，再提出修复方案。

## 代码修改

- 手工编辑文件优先使用 `apply_patch`。
- 不要用临时脚本粗暴改写文件，除非这是明确的批量机械变更。
- 任务信息不足时，涉及安全、数据一致性、不可逆操作或较大范围行为变化的改动应先提出具体问题；低风险局部改动可以说明假设后继续。
- 新增抽象前先确认它满足以下至少一项：
  - 明显降低复杂度；
  - 消除有意义的重复；
  - 符合项目已有架构模式；
  - 让未来变更更容易验证。
- 不要引入新的生产依赖，除非有明确收益；引入前说明原因、替代方案和兼容性。
- 保持错误处理、日志、边界条件和安全约束与项目现有风格一致。
- 新增业务能力时优先保持 handler 薄，把复杂业务逻辑放到独立 package。
- 需要进入 `/readyz` 的外部依赖应实现 `health.Checker`。

## 测试与验证

- 实现新功能、修复 bug、改变行为或重构时，默认先补能失败的测试，再写最小实现。
- 纯文档、纯配置调整可以不强制 TDD，但完成后必须运行合适的验证命令。
- 单元测试应避免依赖正在运行的容器；真实依赖连通性留给 Compose 冒烟验证。
- 常规修改完成后优先运行：

  ```bash
  make check
  ```

- 如果只改 Go 代码，至少运行：

  ```bash
  make fmt-check
  make test
  make vet
  ```

- 如果修改 Compose、环境变量或可观测性配置，至少运行：

  ```bash
  make compose-check
  ```

- 如果无法运行验证命令，最终回复中必须说明原因和剩余风险。

## 调试流程

- 遇到失败时先复现，再定位原因，不要猜测式修复。
- 区分症状、根因和修复手段。
- 优先使用最小复现、日志、测试和调用链分析验证判断。
- 修复后必须重新运行能证明问题已解决的验证命令。

## 安全与隐私

- 不要泄露密钥、令牌、凭据、私有数据或内部链接。
- 不要把敏感信息写入日志、测试快照、提交信息或文档。
- 不要提交真实 `.env`。
- 涉及权限、认证、支付、数据删除、外部请求、文件系统访问时，优先检查安全边界。
- 不要运行破坏性命令，例如 `git reset --hard`、大范围删除文件、强制覆盖远程状态，除非用户明确要求。

## Git 与协作

- 修改前注意工作区可能已有用户改动，不要擅自回滚。
- 不要擅自提交、推送、发布、创建 PR 或修改远程资源，除非用户明确要求。
- 如果需要提交，按逻辑分组暂存文件，不使用 `git add .` 盲目提交。
- 提交信息应简洁描述变更范围和目的。

## 工作汇报

- 简单任务直接执行并给出结果。
- 复杂或高风险任务先给出简短计划，再实施。
- 长时间任务中定期说明当前发现、正在做什么和下一步。
- 最终回复应包含修改内容、验证命令和剩余风险。
